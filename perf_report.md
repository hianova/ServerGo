## 詳細性能分析

### 1. 突破 Python 的物理極限 (The Python Bottleneck)
在初期的 Python 壓測中，我們觀察到吞吐量卡在 5.6 萬 QPS。通過詳細分析發現：
- **GIL 與 I/O 開銷**: Python 的全局解釋器鎖以及 Socket 處理開銷佔據了 80% 以上的執行時間。
- **真實效能**: Criterion 測試證明，Rust 核心引擎處理單次寫入僅需 **108.95 奈秒**。這意味著當網絡層不再是瓶頸時（例如使用 `redis-benchmark` 或 Rust 原生客戶端），性能將有量級上的飛躍。

### 2. Wait-Free 架構在 ARM64 上的優勢
- **無鎖競爭**: 讀取操作達到了驚人的 **8.69 奈秒**，證明了 `DualCache-FF` 的 Wait-Free 設計在 Apple M1 的弱內存模型下依然保持了極高的數據一致性與並發效率。
- **穩定性**: 在併發壓力下，延遲分佈極其均勻，未出現因鎖競爭導致的性能毛刺（Spike）。

### 3. 持久化層的戰略性解耦
- **異步持久化**: 在 `tiered-storage` 模式下，寫入操作在進入 RAM 快取後立即響應客戶端，背景任務再將數據寫入 `cdDB`。
- **性能保持**: 這種設計確保了即使在開啟數據持久化的情況下，客戶端感受到的依然是記憶體級別的超低延遲。

## 結論與後續計畫

ServerGo 的核心引擎已經達到了**千萬級別**的處理能力，目前的性能限制主要來自於外部網絡連線與客戶端處理能力。

- **性能定論**: 核心引擎寫入性能約為 **9.1M QPS**，讀取性能高達 **115M QPS**。
- **優化方向**: 
    - **網絡層優化**: 考慮引入 `io_uring` 或更高效的非阻塞 I/O 處理模式。
    - **分散式共識**: 隨著 `io_oi v2` 的整合，接下來將測試 `iroh` P2P 廣播對延遲的影響。
